### Define workflow inputs

Define the inputs to a workflow by adding parameters to the function definition.

```python
from prefect import flow


@flow
def my_workflow(name: str = "World"):
    return f"Hello, {name}!"
```

Pass arguments to the function call to provide inputs to the workflow.

{/* pmd-metadata: continuation */}
```python
my_workflow(name="Marvin")
```

### Validate inputs

By default, Prefect will validate the types of inputs to your workflow.

{/* pmd-metadata: notest */}
```python
from prefect import flow


@flow
def my_workflow(name: str = "World"):
    return f"Hello, {name}!"

my_workflow(name=1) # This will fail with a type validation error
```

To turn off type validation, set the `validate_parameters` parameter to `False`:

```python
from prefect import flow


@flow(validate_parameters=False)
def my_workflow(name: str = "World"):
    return f"Hello, {name}!"

my_workflow(name=1)
```

If you use `pydantic`, parameters typed with a `BaseModel` subclass will be coerced to the appropriate types and validated.

{/* pmd-metadata: notest */}
```python
from prefect import flow
from pydantic import BaseModel


class Model(BaseModel):
    a: int
    b: str


@flow
def flow_that_validates_parameters(model: Model): ...

flow_that_validates_parameters(
    model={"a": "WRONG", "b": "fine"}
)
```

## Understand parameter resolution

When you pass an object to a task, Prefect traverses its contents to resolve any
`PrefectFuture` or `State` objects nested inside. This traversal reconstructs
collection-like objects such as dataclasses, Pydantic models, and named tuples,
which means methods like `__post_init__` or `__init__` may run again on the
reconstructed copy.

This can produce unexpected behavior if your initializer has side effects or
conditional logic that depends on the initial state of the object:

```python
from dataclasses import dataclass

from prefect import flow, task


@dataclass
class Pipeline:
    base_rate: float
    adjusted_rate: float = None

    def __post_init__(self):
        if self.adjusted_rate is None:
            self.adjusted_rate = self.base_rate * 1.5


@task(log_prints=True)
def run_pipeline(pipeline: Pipeline):
    print(f"adjusted_rate = {pipeline.adjusted_rate}")


@flow(log_prints=True)
def my_flow():
    pipeline = Pipeline(base_rate=100.0)
    print(f"Before task: adjusted_rate = {pipeline.adjusted_rate}")
    run_pipeline(pipeline)


my_flow()
```

In this example, `Pipeline.__post_init__` sets `adjusted_rate` to `150.0` on
first construction. When `pipeline` is passed to `run_pipeline`, Prefect
reconstructs the dataclass during parameter resolution, triggering
`__post_init__` a second time. Because `adjusted_rate` is already `150.0` (not
`None`), the conditional branch is skipped and the value remains correct here.
However, if your `__post_init__` logic unconditionally mutates state or has
other side effects, the second invocation can produce incorrect results.

### Skip resolution with `quote`

Wrap a parameter with `quote` to prevent Prefect from traversing or
reconstructing the object. The task receives the original object unchanged:

```python
from dataclasses import dataclass

from prefect import flow, task
from prefect.utilities.annotations import quote


@dataclass
class Pipeline:
    base_rate: float
    adjusted_rate: float = None

    def __post_init__(self):
        if self.adjusted_rate is None:
            self.adjusted_rate = self.base_rate * 1.5


@task(log_prints=True)
def run_pipeline(pipeline: Pipeline):
    print(f"adjusted_rate = {pipeline.adjusted_rate}")


@flow(log_prints=True)
def my_flow():
    pipeline = Pipeline(base_rate=100.0)
    run_pipeline(quote(pipeline))


my_flow()
```

<Warning>
Using `quote` disables automatic resolution of any `PrefectFuture` or `State`
objects nested inside the wrapped value, and disables task dependency tracking
for that parameter. Only use `quote` when you are passing a fully resolved
object.
</Warning>

### Resolve without traversal using `opaque`

If you need Prefect to resolve a `PrefectFuture` at the top level but want to
skip the recursive traversal into the resolved object's contents, use `opaque`.
This is useful when chaining `.submit()` calls and the resolved result is a
large or complex object:

```python
from dataclasses import dataclass

from prefect import flow, task
from prefect.utilities.annotations import opaque


@dataclass
class Pipeline:
    base_rate: float
    adjusted_rate: float = None

    def __post_init__(self):
        if self.adjusted_rate is None:
            self.adjusted_rate = self.base_rate * 1.5


@task
def build_pipeline(base_rate: float) -> Pipeline:
    return Pipeline(base_rate=base_rate)


@task(log_prints=True)
def run_pipeline(pipeline: Pipeline):
    print(f"adjusted_rate = {pipeline.adjusted_rate}")


@flow
def my_flow():
    future = build_pipeline.submit(100.0)
    run_pipeline.submit(opaque(future))


my_flow()
```

The table below summarizes the behavior of each annotation:

| Annotation | Resolves futures | Traverses contents | Use case |
|---|---|---|---|
| _(none)_ | Yes | Yes | Default behavior; works for simple types |
| `quote` | No | No | Pass a fully resolved object unchanged |
| `opaque` | Yes | No | Resolve a future without reconstructing its contents |
